# ADR-010: Infraestrutura de Plugins — Interfaces Compartilhadas, Testes e Health Check

| Campo | Valor |
|-------|-------|
| **ID** | ADR-010 |
| **Data** | 2026-05-15 |
| **Status** | 🟢 CONFIRMADO |
| **Decisão** | Consolidar interfaces compartilhadas, testes unitários e health check runtime para plugins OpenCode |
| **Alternativas** | Manter interfaces duplicadas em cada plugin, sem teste formal, sem health check |

## Contexto

O ecossistema OpenCode possui dois plugins TypeScript nativos (ecosystem-sync.ts e manus-evolve.ts)
que compartilham padrões comuns: tipos de health score, métricas de ferramentas, estado de eficiência de tokens
e I/O de estado em diretório .evolve/. Antes desta ADR, cada plugin definia suas próprias interfaces
localmente, levando a duplicação e risco de divergência.

Além disso, não existiam testes unitários formais para plugins — apenas testes de integração no core/.

## Decisão

1. **Interface compartilhada** (plugins/interfaces.ts):
   - Extrair ComponentHealth, ToolMetric, SessionMetrics, TokenEfficiencyState, EvolutionRound
     de ambos os plugins
   - Exportar helpers: scoreToStatus(), createSessionMetrics(), writePluginState(), eadPluginState()
   - Manter import type { Plugin } from "@opencode-ai/plugin" como única dependência externa

2. **Testes unitários** (plugins/plugin.test.ts):
   - Usar un test como test runner (já disponível no ambiente v1.3.13)
   - Testar cada interface helper isoladamente
   - Testar carregamento de módulo de cada plugin
   - Testar que hooks principais existem e não lançam exceções
   - Mockar Plugin context com objetos mínimos

3. **Health check runtime** (plugins/health.ts):
   - CLI scan que lê .evolve/ecosystem-state.json e reporta saúde geral
   - Modo --watch para re-check a cada 60s
   - Fallback para scan do diretório quando state não existe
   - Relatório JSON tipado exportado como HealthReport

4. **Validação cíclica**: Extraída de ecosystem-sync.ts → interfaces.ts helpers
   (scoreToStatus, computeCrossValidation permanece no plugin por ser específica)

## Consequências

- **Positivas**: Um arquivo de interfaces para manter, em vez de dois (DRY)
- **Positivas**: Testes detectam regressões nos hooks dos plugins
- **Positivas**: Health check permite diagnóstico rápido via CLI
- **Negativas**: Plugins precisam ser refatorados para importar de ./interfaces
- **Mitigação**: A refatoração é puramente cosmética (renomear tipos), sem mudança de comportamento

## Confiança

| Aspecto | Confiança |
|---------|-----------|
| Interfaces extraídas corretamente | 🟢 CONFIRMADO — análise manual dos dois plugins |
| Testes cobrem hooks principais | 🟢 CONFIRMADO — 11 testes definidos |
| Health check funcional | 🟡 INFERIDO — depende de estado persistido |
| Refatoração dos plugins para usar interfaces | 🔴 LACUNA — requer validação em runtime |

## Artefatos

- plugins/interfaces.ts — Interfaces compartilhadas + helpers
- plugins/plugin.test.ts — Testes unitários (11 testes)
- plugins/health.ts — Health check runtime
